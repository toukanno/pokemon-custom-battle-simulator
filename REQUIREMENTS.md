# Pokemon Custom Battle Simulator 用件定義

## 1. 目的
本システムは、ポケモン対戦の基本メカニクス（タイプ相性・ダメージ計算・技管理・チーム構築）をプログラム上で再現し、
テスト可能なライブラリとして提供する。

## 2. スコープ
- Node.js 環境で動作するライブラリとして提供する。
- 1対1のターンベースバトル処理を対象とする。
- 既存の図鑑データと技データを用いた、ポケモン生成・チーム生成を対象とする。

### 2.1 スコープ外
- GUI/フロントエンド画面
- ネットワーク対戦
- 全世代・全ポケモン・全技の完全収録

## 3. 想定ユーザー
- 対戦ロジックを検証したい開発者
- ユニットテストでゲームロジック品質を担保したいチーム
- シミュレーション学習用途の利用者

## 4. 機能要件

### FR-01: ポケモン生成
- 種族値・タイプ・レベル・性格・個体値(IV)・努力値(EV)・技構成からポケモンを生成できること。
- 実数値（HP/攻撃/防御/特攻/特防/素早さ）を自動計算できること。

### FR-02: ステータス管理
- 現在HP、状態異常、能力ランク、生存状態を保持できること。
- ダメージ適用・回復・状態異常付与・能力変化を処理できること。

### FR-03: 技管理
- 技IDから技情報を取得できること。
- 1体あたりの技数制約（最大4）を検証できること。
- 登録されていない技IDを無効として扱えること。

### FR-04: タイプ相性
- 攻撃タイプと防御側タイプ配列から相性倍率を算出できること。
- 効果メッセージ（こうかなし/いまひとつ/ばつぐん）を取得できること。

### FR-05: ダメージ計算
- 物理・特殊カテゴリに応じたダメージ計算ができること。
- タイプ一致補正(STAB)、タイプ相性、急所、乱数、最小ダメージ制約を反映できること。
- 反動ダメージ・こんらん自傷ダメージを計算できること。

### FR-06: バトル進行
- 2体のポケモンでターンを進行できること。
- 行動順判定（素早さ比較）と技使用結果を処理できること。
- ひんし判定と勝敗判定ができること。

### FR-07: チームビルド
- 図鑑データからチーム（1〜6体）を生成できること。
- チームバリデーション（体数、EV合計、EV上限、IV範囲、レベル範囲）を実施できること。

### FR-08: 公開API
- 下記の主要APIを外部公開すること。
  - `Pokemon`, `Battle`
  - 技関連: `MOVES`, `getMoveById`, `validateMoveset`
  - タイプ関連: `TYPES`, `getEffectiveness`, `getEffectivenessMessage`
  - ダメージ関連: `calculateDamage`, `getCriticalHitChance`
  - チーム関連: `POKEDEX`, `buildPokemon`, `buildTeam`, `validateTeam`

## 5. 非機能要件

### NFR-01: テスト容易性
- コアロジックは関数/クラスとして分離し、ユニットテストで検証可能であること。

### NFR-02: 保守性
- ドメイン別（pokemon, battle, damage, moves, types, teambuilder）にモジュール分割すること。

### NFR-03: 再利用性
- `src/index.js` から集約エクスポートし、外部利用時の導線を単純化すること。

### NFR-04: 実行環境
- Node.js + npm でセットアップ・テスト実行可能であること。

## 6. 制約・前提
- データセット（図鑑・技）は実装済み範囲を正とする。
- ルールは簡略化されており、公式対戦仕様の全要素を網羅しない。

## 7. 受け入れ基準（例）
- `npm test` が成功すること。
- 既存テスト（types, damage, pokemon, moves, teambuilder）がすべて成功すること。
- `buildTeam` が不正入力に対して `null` を返し、`validateTeam` がエラー理由を返せること。
- タイプ相性 0 倍時にメッセージが「It had no effect!」となること。
